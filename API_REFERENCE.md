# Vera Bot API Reference

Quick reference for all 5 API endpoints.

## Base URL

```
http://localhost:5000
```

---

## 1. POST /v1/context

**Purpose**: Receive context updates (category, merchant, customer, trigger)

**Request**:
```bash
curl -X POST http://localhost:5000/v1/context \
  -H "Content-Type: application/json" \
  -d '{
    "scope": "category|merchant|customer|trigger",
    "context_id": "unique_id",
    "version": 1,
    "payload": { /* context object */ },
    "delivered_at": "2026-04-26T10:00:00Z"
  }'
```

**Success Response (200)**:
```json
{
  "accepted": true,
  "ack_id": "ack_abc123",
  "stored_at": "2026-04-26T10:00:00.123Z"
}
```

**Version Conflict (409)**:
```json
{
  "accepted": false,
  "reason": "stale_version",
  "current_version": 5
}
```

**Invalid Request (400)**:
```json
{
  "accepted": false,
  "reason": "invalid_scope",
  "details": "scope must be one of [category, merchant, customer, trigger]"
}
```

**Scopes**:
- `category` — Category context (dentists, salons, restaurants, etc.)
- `merchant` — Merchant business state
- `customer` — Customer relationship data
- `trigger` — Event triggers (research, recall, performance, festival, etc.)

**Behavior**:
- Idempotent by (context_id, version)
- Higher version replaces lower version
- Stores context until test ends

## 2. POST /v1/tick

**Purpose**: Periodic wake-up; bot decides whether to initiate proactive messages

**Request**:
```bash
curl -X POST http://localhost:5000/v1/tick \
  -H "Content-Type: application/json" \
  -d '{
    "now": "2026-04-26T10:30:00Z",
    "available_triggers": [
      "trg_2026_04_26_research_digest",
      "trg_2026_04_26_recall_priya"
    ]
  }'
```

**Response (200)**:
```json
{
  "actions": [
    {
      "conversation_id": "conv_abc123",
      "merchant_id": "m_001_drmeera",
      "customer_id": null,
      "send_as": "vera",
      "trigger_id": "trg_2026_04_26_research_digest",
      "template_name": "vera_research_digest_v1",
      "template_params": ["Dr. Meera", "JIDA Oct", "..."],
      "body": "Dr. Meera, JIDA's Oct issue...",
      "cta": "binary_yes_no",
      "suppression_key": "research:dentists:2026-W17",
      "rationale": "External research digest with merchant-relevant clinical anchor..."
    }
  ]
}
```

**Fields**:
- `conversation_id`: Unique ID for this conversation (generated by bot)
- `send_as`: Who's sending — `"vera"` or `"merchant_on_behalf"`
- `cta`: Call-to-action type — `"binary_yes_no"`, `"open_ended"`, or `"none"`
- `body`: The exact WhatsApp message text
- `rationale`: Explanation of why this message works (compulsion lever + audience insight)

**Behavior**:
- Returns array of actions (can be empty if no triggers ready)
- Each action creates a new conversation
- Suppression key prevents duplicate messages
- Bot responds within 30 seconds

## 3. POST /v1/reply

**Purpose**: Handle incoming reply from merchant or customer

**Request**:
```bash
curl -X POST http://localhost:5000/v1/reply \
  -H "Content-Type: application/json" \
  -d '{
    "conversation_id": "conv_abc123",
    "merchant_id": "m_001_drmeera",
    "customer_id": null,
    "from_role": "merchant",
    "message": "Yes, send me the abstract",
    "received_at": "2026-04-26T10:45:00Z",
    "turn_number": 2
  }'
```

**Response (200) — Send Next Message**:
```json
{
  "action": "send",
  "body": "Shukriya Dr. Meera! 🙌 Sending now...",
  "cta": "open_ended",
  "rationale": "Honoring the merchant's accept; adding next-best-step..."
}
```

**Response (200) — Wait**:
```json
{
  "action": "wait",
  "wait_seconds": 1800,
  "rationale": "Merchant asked for time; backing off 30 minutes"
}
```

**Response (200) — End Conversation**:
```json
{
  "action": "end",
  "rationale": "Merchant said not interested; gracefully exiting"
}
```

**Actions**:
- `send` — Compose and send next message (include `body` + `cta`)
- `wait` — Back off for specified seconds
- `end` — End conversation gracefully

**Behavior**:
- Bot must respond within 30 seconds
- Detects intent (positive/negative/delay/auto-reply)
- Maintains conversation state
- One response per reply

## 4. GET /v1/healthz

**Purpose**: Liveness probe (judge polls every 60 seconds; 3 consecutive failures = disqualified)

**Request**:
```bash
curl http://localhost:5000/v1/healthz
```

**Response (200)**:
```json
{
  "status": "ok",
  "uptime_seconds": 3600,
  "contexts_loaded": {
    "category": 5,
    "merchant": 50,
    "customer": 200,
    "trigger": 100
  }
}
```

**What It Means**:
- `status: "ok"` — Bot is alive
- `uptime_seconds` — Seconds since startup
- `contexts_loaded` — Count of stored contexts per scope

## Trigger Kinds

| Kind | Scope | Example |
|------|-------|---------|
| `research_digest` | merchant | Latest industry research + citations |
| `regulation_change` | merchant | Compliance deadline (DCI, JIDA) |
| `recall_due` | customer | Appointment reminder (6-month cleaning) |
| `perf_dip` | merchant | Performance dropped (calls, CTR) |
| `renewal_due` | merchant | Subscription expiring soon |
| `festival_upcoming` | merchant | Diwali, wedding season prep |
| `wedding_package_followup` | customer | Bridal package after trial |

## CTA Types

| CTA | Format | Example |
|-----|--------|---------|
| `binary_yes_no` | Expects "Yes" or "No" | "Want me to draft 3 posts?" → "Reply YES" |
| `open_ended` | Any response | "What else would help?" → Merchant can reply freely |
| `none` | No response expected | FYI message only |

## Common Error Scenarios

### Context not stored
```bash
curl -X POST http://localhost:5000/v1/context \
  -d '{"scope": "invalid", ...}'
# → 400 Bad Request
# → {"accepted": false, "reason": "invalid_scope", ...}
```

### Version conflict
```bash
# Store v1
curl -X POST http://localhost:5000/v1/context \
  -d '{"scope": "merchant", "context_id": "m_001", "version": 1, ...}'
# → 200 OK

# Try to store v0 (older)
curl -X POST http://localhost:5000/v1/context \
  -d '{"scope": "merchant", "context_id": "m_001", "version": 0, ...}'
# → 409 Conflict
# → {"accepted": false, "reason": "stale_version", "current_version": 1}
```

### Bot timeout
If `/v1/reply` takes > 30 seconds, judge marks turn as timeout and proceeds.

## Performance Notes

- **Response time**: < 1 second (health check)
- **Message composition**: 5-10 seconds (Claude API + context processing)
- **Conversation state**: In-memory, O(1) lookup
- **Suppression**: O(1) check per trigger
- **Maximum conversations**: Limited by memory (typical: 1000+ concurrent)

---

## Security Notes

- ✅ No secrets exposed in responses
- ✅ API key stored in environment variable only
- ✅ HTTP (not HTTPS) for local testing; use HTTPS in production
- ✅ All inputs validated before processing
- ✅ No SQL injection (in-memory, no DB)
- ✅ No XSS (JSON-only, no HTML rendering)

## Debugging Tips

### Enable verbose logging
```python
# In bot.py, change logging level:
logging.basicConfig(level=logging.DEBUG)
```

### Check context loaded
```bash
curl http://localhost:5000/v1/healthz | jq '.contexts_loaded'
# {"category": 5, "merchant": 50, "customer": 200, "trigger": 25}
```

### Inspect conversation state
```python
# In bot.py REPL:
from bot import store
conv = store.get_conversation("conv_abc123")
print(json.dumps(conv, indent=2))
```

### Test Claude integration
```bash
# If Claude call fails:
export ANTHROPIC_API_KEY="test-invalid-key"
```
